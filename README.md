# airwindohhs

[Airwindows](https://github.com/airwindows/airwindows) VST plugins modified as headers you can include in any C++ project.

## Build

Consume via CMake's FetchContent:

```cmake
include(FetchContent)
FetchContent_Declare(
    airwindohhs
    GIT_REPOSITORY https://github.com/isabelgk/airwindohhs.git
    GIT_TAG main
)
FetchContent_MakeAvailable(airwindohhs)

target_link_libraries(your_target PRIVATE Airwindohhs::Airwindohhs)
```

`Airwindohhs` is a header-only `INTERFACE` library requiring C++17, propagated
automatically to consumers. `airwindows-version.txt` at the repo root pins the upstream
`airwindows/airwindows` commit the headers were generated from. 

## Usage

Every plugin is a header under `include/<category>/<plugin>.hpp`, in namespace
`airwindohhs::<plugin>`, categorized to match
[Airwindopedia](https://github.com/airwindows/airwindows)'s grouping (dynamics,
reverb, tape, etc).

```cpp
#include "dynamics/buttercomp.hpp"

airwindohhs::buttercomp::ButterComp<float> comp;
comp.setSampleRate(44100.0f);
comp.set_parameter_value(airwindohhs::buttercomp::ButterComp<float>::kParamA, 0.5f);
comp.process(inputs, outputs, sampleFrames);
```

## Meta plugin

`include/meta.hpp` is a single header giving programmatic access to every
plugin in the repo: their names, categories, tags, and descriptions, plus a
host object that can load and run any of them interchangeably at runtime.

```cpp
#include "meta.hpp"

airwindohhs::meta::PluginHost<float> host;

for (const auto& entry : host.all_plugins()) {
    std::cout << entry.category << "/" << entry.slug << ": " << entry.name << "\n";
}

host.load_by_name("ButterComp");
host.setSampleRate(44100.0f);
host.set_parameter_value(0, 0.5f);
host.process(inputs, outputs, sampleFrames);

host.load_by_index(3); // swap to a different algorithm; sample rate carries over
```

`PluginEntry<T>::make` and `load_by_index`/`load_by_name` construct a
`PluginAdapter<T, PluginClass>`, a generic wrapper implementing the
`IPlugin<T>` interface, so any plugin can be reached through the same handful
of virtual calls regardless of its own parameter count or internals.

This pulls in all plugin headers which is a large compile-time cost. Prefer
including a plugin directly (see Usage above) unless you actually need runtime
discovery or switching.

`include/meta/registry.hpp` is generated by `scripts/gen_meta.py`, which
rescans `include/**/*.hpp` and rewrites the registry from scratch, so it's
always regenerated in full rather than updated incrementally:

```
python3 scripts/gen_meta.py
```

CI regenerates it and fails if the checked-in copy doesn't match, so it's
always in sync with whatever's actually in `include/`.

## Generating headers

Everything under `include/` is generated from the upstream Airwindows VST
source by `scripts/grab`, a small [uv](https://docs.astral.sh/uv/)-managed CLI.
It parses each plugin's `.cpp`/`.h` source with libclang and fills in
`scripts/res/template.hpp` per plugin.

```
uv run grab
```

By default this clones the upstream `airwindows/airwindows` repo (shallow,
sparse-checked-out to just the plugin sources and `Airwindopedia.txt`) at the
commit pinned in `airwindows-version.txt`, and regenerates every header under `include/`.

Run `uv run grab --help` for the full list of flags.

Requires a `clang`/`clang++` on `PATH`; the generator shells out to it once
per run to locate system header search paths for libclang, which ships without
any headers of its own.
